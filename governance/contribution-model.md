# Contribution Model

## Purpose

This document explains how contributors should work in the repository.

Its purpose is to keep the repository coherent as a public architecture accelerator rather than letting it drift into conflicting contributions.

## Contribution Expectations

Contributions should improve at least one of the following:

- architecture clarity
- governance clarity
- template quality
- example quality
- AI-assisted engineering quality
- machine-readable architecture quality

## What Contributors Must Respect

Contributors must respect:

- the glossary
- the hub-and-spoke overview
- the environment model
- the mandatory platform standards
- the Medallion ownership model
- the public-template safety rules

## How to Propose Architecture Changes

If a contribution changes architecture behavior, the contributor must:

1. identify the standard or boundary being changed
2. explain why the change is needed
3. update all affected files, not only one document
4. make the change explicit in the pull request

## Reviewability Rule

Every contribution should be easy to review.

That means:

- clear scope
- clear rationale
- explicit affected files
- explicit architecture impact

## Public-Safe Rule

Do not add:

- internal company URLs
- internal account identifiers
- confidential operating procedures
- enterprise-specific private assumptions

## AI-Assisted Contribution Rule

If AI is used to generate content, the result must still be reviewed against the repository architecture. AI-generated text or code is not acceptable just because it is fluent.
